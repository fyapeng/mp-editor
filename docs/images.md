# 正文图片指南

## 一、当前可以怎样插图

网页编辑器支持：

- 点击 Markdown 工具栏的“图片”；
- 把图片拖入编辑区；
- 在编辑区直接粘贴截图或剪贴板图片。

浏览器会把图片转换成 Data URL，并写入 Markdown：

```markdown
![图示说明](data:image/png;base64,...)
```

这种方式不需要图床，适合本地预览和临时排版。不过 Data URL 会增大 Markdown 和浏览器本地存储，粘贴到公众号后也不保证始终被保留，因此不应作为长期发布链路。

## 二、为什么很多编辑器要求图床

Markdown 中的图片最终会变成：

```html
<img src="图片地址">
```

公众号后台需要能够读取这个地址。`C:\图片\figure.png` 只存在于作者电脑上，其他设备无法访问；Data URL 又可能在粘贴清洗过程中被过滤。图床把图片变成公开的 HTTPS 地址，因此复制富文本时更稳定。

## 三、MP Editor 推荐的方案

推荐优先使用“微信自身作为图片托管”：

1. 用户在编辑器中选择本地图片；
2. 本地助手读取图片；
3. 本地助手调用公众号正文图片上传接口；
4. 微信返回图片 URL；
5. 编辑器把正文 `<img src>` 替换为该 URL；
6. 再生成或更新草稿。

这样不需要额外注册第三方图床，图片也由目标公众号的微信素材链路托管。由于上传需要 AppSecret，这一步只能在本地助手中执行，不能放在公开的 GitHub Pages 前端。

## 四、其他可选方案

| 方案 | 适用场景 | 注意事项 |
| --- | --- | --- |
| 微信正文图片上传 | 直接写入公众号草稿 | 推荐；需要公众号接口权限和本地助手 |
| 自建对象存储 | 多平台复用图片 | 需要域名、HTTPS、费用和访问策略 |
| GitHub 仓库或 Pages | 少量公开图片 | 不适合大量图片和频繁替换 |
| 第三方图床 | 快速开始 | 需评估稳定性、隐私、限额和迁移成本 |
| Data URL | 本地预览 | 文件膨胀，公众号兼容性不稳定 |

## 五、建议的图片资源管理

后续版本将采用以下规则：

- 原稿使用相对路径，例如 `assets/figure-01.jpg`；
- 导入 Markdown 时同时选择文章资源目录，或导入包含 Markdown 与图片的 ZIP；
- 上传前自动压缩过大的图片并保留原图；
- 支持替换、删除、排序、替代文字和图注；
- 对每张图片计算哈希，避免重复上传；
- 草稿预览明确标出“本地图片”“外部图片”和“微信已托管图片”。

在正文图片自动上传功能完成前，最稳妥的发布方式是：先复制文字与公式排版，再在公众号后台插入正文图片；或者自行使用可靠图床后，在 Markdown 中填写 HTTPS 图片地址。
